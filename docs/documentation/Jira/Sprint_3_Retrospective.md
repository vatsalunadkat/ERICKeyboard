# ERICK — Sprint 3 Retrospective

**Sprint**: SCRUM Sprint 3  
**Dates**: March 9–13, 2026 (Monday–Friday)
**Velocity**: ~67 story points planned  
**Sprint Goal**: Complete and publicly release the Android keyboard; finalize core keyboard features on both platforms  

## 👍 What Went Well

1. **Core deliverables completed** — All core commitments for prototype and keyboard development were fulfilled. The Android keyboard was fully developed and ready for public release by sprint end. Overall core development tasks were implemented smoothly.

2. **Clear requirement definitions** — Jira tasks were clearly defined and well-organized, containing explicit expected requirements and acceptance criteria. Team members could quickly clarify their work content and reduce execution deviations.

3. **Strong agile teamwork** — Members helped each other effectively. When encountering technical issues (such as Android Studio and Room configuration problems), teammates provided prompt assistance, allowing issues to be resolved quickly. Daily stand-ups ran smoothly, with all members synchronizing their work progress, making it easy to identify and help resolve bottleneck issues.

4. **Team growth and skill improvement** — Each member improved their proficiency in their respective areas and gained a clearer understanding of the overall development process and architecture. New members adapted quickly to the work rhythm.

5. **Independent tasks progressed efficiently** — Some tasks had low coupling with other team work, enabling efficient parallel progress. Work order allocation was reasonable, allowing members to focus on their own modules.

6. **Milestone achieved** — Project work order management surpassed 100 work items, with nearly 60 development tasks completed. Project progress showed clear quantifiable results.

---

## 👎 What Didn't Go Well

1. **Difficult task integration** — Each team member's tasks were relatively independent, which led to a significant amount of time and energy being spent on integrating all content in the later stage. Integration became the main bottleneck in this iteration.

2. **Insufficient testing** — The game development portion was completed by multiple members, with many potential edge-case issues and vulnerabilities (such as incorrect machine allocation when multiple people input simultaneously). These issues could not be found by reviewing code alone and could only be exposed through actual testing.
